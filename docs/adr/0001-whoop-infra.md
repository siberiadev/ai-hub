# ADR 0001 — Инфраструктура WHOOP-интеграции

- **Статус:** принято (2026-06-25)
- **Контекст:** добавляем интеграцию WHOOP → БД → Claude (см. [whoop-integration-plan.md](../whoop-integration-plan.md)).
  Фаза 0 фиксирует архитектурные решения, на которые опираются все последующие фазы.
- **Исходные ограничения проекта:** приложение слушает только `127.0.0.1` (бот на long-polling,
  входящих портов нет), ufw пускает лишь SSH; данные сессий лежат в локальном SQLite
  (`data/ai-hub.db`, `better-sqlite3` в `SessionService`); режим `bypassPermissions` по умолчанию
  даёт Claude фактический шелл на сервере — публичный периметр надо держать минимальным.

---

## Решение 1 — Публичный ingress: Cloudflare Tunnel

`cloudflared` на дроплете держит **исходящее** соединение к Cloudflare; входящие порты не открываем
(ufw остаётся `OpenSSH only`). Cloudflare проксирует `https://<host>/whoop/*` → `http://127.0.0.1:3000`.

**Почему:** сохраняет текущую модель «наружу ничего не торчит» (нулевой входящий периметр), не требует
открывать 80/443 и держать TLS-терминатор. Даёт стабильный hostname под регистрацию в WHOOP Dashboard.

**Следствия:** нужен домен, добавленный в Cloudflare. `main.ts` не меняется (листен остаётся `127.0.0.1`).
Альтернатива (Caddy + Let's Encrypt) отклонена: расширяет периметр (открытые 80/443) без выгоды для
сценария с одним пользователем.

## Решение 2 — БД для данных WHOOP: внешний Postgres (`DATABASE_URL` + SSL)

WHOOP-данные хранятся во **внешнем managed Postgres**; TypeORM подключается драйвером `pg` с включённым
`ssl`. Sessions-БД (`SessionService`, SQLite) **остаётся без изменений**.

**Почему:** две независимые подсистемы — никакой конкуренции за блокировки SQLite; бэкап WHOOP-данных
обеспечивает managed-провайдер; чистая граница миграций. MCP сможет читать ту же БД отдельным
read-only подключением.

**Следствия:** новые зависимости `@nestjs/typeorm typeorm pg` (ставятся в Фазе 1). Нужна аккуратная
настройка SSL под провайдера (часто `ssl: { rejectUnauthorized: false }` или явный CA).

## Решение 3 — Шифрование токенов: AES-256-GCM

OAuth access/refresh токены хранятся в БД в зашифрованном виде. Алгоритм AES-256-GCM на встроенном
`node:crypto`, ключ из `WHOOP_TOKEN_ENC_KEY` (32 байта base64).

**Почему:** дешёвая защита-в-глубину, в первую очередь для бэкапов и сценария утечки **только** БД
(ключ живёт в `.env` с `chmod 600`, не в БД). Без новых зависимостей.

**Следствия:** при утрате `WHOOP_TOKEN_ENC_KEY` потребуется повторный OAuth. Ротация ключа — отдельная
процедура (вне Фазы 0).

## Решение 4 — MCP-процесс: отдельный stdio-энтрипоинт, Postgres read-only

MCP-сервер `whoop` — отдельный stdio-процесс (запускается Claude CLI), подключается к Postgres
**только на чтение** через отдельную роль / `WHOOP_MCP_DATABASE_URL`.

**Почему:** изоляция прав (Claude физически не может писать в БД через MCP), независимый деплой/перезапуск.

**Следствия:** нужна read-only роль в Postgres (`GRANT SELECT`). Реализация тулзы — Фаза 6; в Фазе 0
только заводим переменную и роль.
